## Question stack : PHP vs JS ?

Discussion du 13/05. Serge se demande si PHP limite le rendu "moderne".

Conclusion : non. Le feeling "pas moderne" vient du design (formulaires classiques, textareas brutes, rechargements), pas de la techno. PHP génère le HTML, le frontend est libre (Tailwind, JS vanilla, libs). Stripe, WordPress, Laravel = PHP et personne ne trouve ça vieux.

Passer en React/Next.js apporterait des interactions plus fluides nativement mais ajouterait : un hébergement Node (Vercel ou VPS), un build system, une complexité de déploiement. O2switch ne fait pas de Node. Pour 4-5 clients, c'est de l'over-engineering.

Le vrai levier : investir dans le design du Point Hebdo. En faire une page type Notion (blocs éditables, drag-and-drop actions via Sortable.js, timeline visuelle) plutôt qu'un formulaire admin. Tout ça se fait en JS vanilla sur la stack actuelle.

Décision : on reste en PHP + Tailwind + JS vanilla. Prochaine session = focus UX/design du Point Hebdo pour en faire une page montrable à un client.

## Vision élargie (discussion 13/05, fin de session)

Serge veut que l'espace client devienne aussi son outil d'administration des missions (côté admin uniquement). Pas un remplacement du cockpit global, mais un vrai tableau de bord opérationnel mission.

Ce qui est dans le périmètre espace client :
- Dashboard admin multi-missions (vue d'ensemble : phase, actions en attente, météo, prochains jalons)
- Gantt par mission (timeline visuelle des phases, livrables, jalons) — libs JS type Frappe Gantt ou similar
- Actions consultant non visibles par le client (flag visibilité)
- Zone d'échange avec des tiers (intégrateurs, partenaires) — rôle dédié, espace séparé du fil client
- Visuels riches pour le client : graphiques, progressions, données pré-mâchées
- Le client doit se sentir "choyé" — transparence + mise en forme pro

Ce qui reste dans le cockpit Cowork (hors espace client) :
- Prospection, pipeline, valorisation
- Facturation, administratif
- Stratégie personnelle, positionnement
- Planification semaine / multi-projets

La limite : tout ce qui concerne la relation mission (consultant-client-partenaire) va dans l'espace client. Tout ce qui concerne la gestion de l'activité de Serge reste dans le cockpit.

Insight client : les dirigeants PME aiment le visuel, le pré-mâché, le sentiment d'être accompagnés avec rigueur. C'est un argument commercial autant qu'un outil de travail.

## Capitalisation des livrables HTML interactifs

Serge produit des livrables HTML interactifs de très bonne facture (ex: framework gouvernance SI pour Bonnavion — phases, cartes cliquables, timeline, risques/leviers). Ces livrables sont réutilisables d'un client à l'autre.

Stratégie de capitalisation en 3 niveaux :

1. Bibliothèque de templates HTML génériques
   - Framework gouvernance SI (intégration ERP)
   - Diagnostic SI (cartographie, maturité, risques)
   - Carte des motivations dirigeant
   - Grille de sélection ERP / outil
   - Feuille de route SI
   - Chaque template est paramétrable (nom client, contexte, données spécifiques)

2. Intégration dans l'espace client
   - Routage HTML déjà en place (/slug/diagnostics/fichier.html en iframe)
   - Structurer les dossiers missions par type de livrable
   - Référencer dans l'onglet Documents avec type "html"
   - Le client voit un livrable interactif pro, pas un PDF statique

3. Production via Cowork
   - Serge demande "produis le framework gouvernance SI pour [client]"
   - Cowork prend le template + le contexte mission
   - Génère le HTML adapté, prêt à publier
   - Serge valide, uploade, le client le voit
